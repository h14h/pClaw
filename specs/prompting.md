# Prompting System

## Overview

This spec defines a structured system-prompt layer that gives the agent a distinct role/personality
while preserving tool reliability. The design is inspired by OpenClaw's compact sectioned prompts and
mode switching (`full` vs `minimal`).

## Goals

1. Always prepend one `role=system` message to model requests
2. Separate persona content from operational/tool policy
3. Support prompt modes:
   1. `full` for normal inference turns
   2. `minimal` for delegated reasoning paths
4. Keep prompt size bounded and deterministic

## Non-Goals

1. No runtime behavior changes to tool execution in this phase
2. No model selection changes
3. No persistence layer for prompt state

## Prompt Composition

System prompt is assembled from ordered sections:

1. `Identity`: name + role summary
2. `Behavior`: tone and interaction style
3. `Tooling`: how/when to use tools
4. `Safety`: constraints and refusal boundaries
5. `Runtime`: response formatting and environment assumptions

Section order is fixed to reduce drift in model behavior.

## Modes

| Mode | Use Case | Included Sections |
|------|----------|-------------------|
| `full` | Primary model turns (`kimi-k2-instruct`) | Identity, Behavior, Tooling, Safety, Runtime |
| `minimal` | Delegated sub-reasoning (`gpt-oss-120b`) | Behavior, Safety, Runtime (compact) |

`minimal` intentionally excludes persona-heavy flavor text.

## Configuration

Prompt config is represented by `PromptConfig`:

1. `AgentName`
2. `RoleSummary`
3. `Persona`
4. `BehaviorRules`
5. `SafetyRules`
6. `RuntimeRules`
7. `MaxPersonaChars`

Defaults should be deterministic and safe when no external config is provided.

## Injection Rules

Before every chat completion request (inside `withSystemPrompt`):

1. Build system prompt text with `PromptBuilder.Build(mode, context)`
2. Append `[Conversation Summary]` section if a summary is stored in the context (via `conversationSummaryFromContext`)
4. Prepend `ChatMessage{Role: "system", Content: <prompt>}` to conversation
5. Remove any pre-existing leading `system` message generated by prior runtime layers to avoid duplicates

### `[Conversation Summary]` Section

When rolling compaction has produced a summary of older conversation history, `withSystemPrompt` appends it as a `[Conversation Summary]` section **after** `[Memory]` and **before** the conversation messages. This gives the model compressed context for messages that were evicted during compaction.

- Injected only when `conversationSummaryFromContext(ctx)` returns a non-empty string
- Uses `formatSection("Conversation Summary", summary)` for consistent formatting
- Callers set the context value via `withConversationSummary(ctx, cs.Summary)` before each inference call
- Section is omitted entirely when there is no active summary (e.g., early in a conversation)

## Context Inputs

`PromptBuildContext` can include:

1. Registered tool names
2. Transport (`cli` or `discord`)
3. Prompt mode

This allows runtime-aware wording without changing core policy.

## Runtime Split Contract

For transports that split long messages (for example Discord), runtime instructions may ask the model to:

1. Insert `<<MSG_SPLIT>>` between logical sections when output is likely to exceed transport limits
2. Prefer roughly even chunk sizes across splits
3. Avoid leaving a tiny trailing fragment as the final chunk

Split markers are advisory and not guaranteed to appear. Transport adapters must keep deterministic fallback splitting.

## Tool-Call User Messaging

Runtime instructions can require user-facing tool-call transparency:

1. Send one brief status sentence before the first tool call
2. For delegated reasoning, explicitly note that a deeper thinking step is in progress and a complete answer will follow
3. Avoid repeated status chatter when multiple tools are called in one turn

## Validation and Tests

Unit tests should verify:

1. Full mode includes all required section headers
2. Minimal mode omits identity/persona-heavy content
3. Tool names appear in tooling section when supplied
4. Persona text is truncated to configured char limit
5. Prepend helper injects exactly one leading `system` message

## Rollout Plan

1. Phase 1: land interfaces + builder + tests (no inference wiring)
2. Phase 2: wire prompt injection into primary inference and delegated reasoning
3. Phase 3: add env/file configuration for persona profile selection
